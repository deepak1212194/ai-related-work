# Role profile: Software Engineer (Backend / Full-Stack)

For senior IC roles at top-tier product companies. Backend-leaning by default;
the planner can switch to a full-stack bias when the input shows substantial
frontend work.

## priority_keywords

distributed systems, microservices, event-driven, message queue, Kafka,
RabbitMQ, gRPC, REST, GraphQL, OpenAPI, FastAPI, Express, Node.js, Spring,
Go, Java, Python, TypeScript, Rust, C++, PostgreSQL, MySQL, Cosmos DB,
DynamoDB, MongoDB, Redis, ElasticSearch, Kubernetes, Docker, AWS, GCP,
Azure, Lambda, S3, ECS, EKS, CI/CD, GitHub Actions, Jenkins, Argo, Helm,
observability, OpenTelemetry, Prometheus, Grafana, Datadog, SLO, latency,
throughput, sharding, caching, idempotency, retries, circuit breaker,
schema migration, zero-downtime, feature flag, blue-green, canary, A/B,
unit test, integration test, contract test, fuzzing, load test, k6, JMeter.

## hiring_signals

- A named, owned service with traffic numbers (RPS / users / records / GB).
- A migration story (legacy -> new, monolith -> services, on-prem -> cloud).
- An incident the candidate diagnosed and resolved.
- A deliberate trade-off ("chose Kafka over Kinesis because ...").
- Test discipline: a number for coverage or test count is a positive signal.

## red_flags

- "Worked on backend" with no service named.
- Listing every CRUD operation as a separate bullet.
- Frontend-flavoured bullets when the role is backend (plan around it).
- "Familiar with" / "exposure to" — only owned things land.

## seniority_calibration

- L4-L5 (senior): owned service, on-call, ships independently.
- L6 (staff): cross-service architecture, mentorship, design reviews lead.
- L7+ (principal): platform decisions, multi-org impact.
